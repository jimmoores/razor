# KRoC AArch64 Port Fixes

This document tracks the critical fixes implemented during the AArch64 port of KRoC (specifically for Darwin/macOS).

## 1. Compiler (`tranx86`) Fixes

### Workspace Pointer (`x28`) Stability
- **Issue:** Arithmetic and bitwise store instructions were incorrectly using or clobbering `x28` (the workspace pointer).
- **Fix:** Updated `archaarch64.c` to ensure `x28` is strictly preserved and only modified by explicit workspace adjustment instructions (`AJW`).

### Flag-Setting Arithmetic (`adds`, `subs`)
- **Issue:** Loops (like `SEQ i=0 FOR 10`) would only execute once because the loop counter decrement (`sub`) did not set the condition flags.
- **Fix:** Implemented `aarch64_sets_flags` and updated `INS_ADD`, `INS_SUB`, `INS_INC`, and `INS_DEC` to append the `s` suffix when `REG_CC` is present in the output arguments.

### Type Extension Support
- **Issue:** Missing support for zero-extension and sign-extension caused broken `CASE` statement logic in the runtime.
- **Fix:** Implemented `INS_MOVEZEXT8TO32`, `INS_MOVEZEXT16TO32`, and `INS_MOVESEXT16TO32` using `ldrb`, `ldrh`, and `ldrsh`.

### Byte Operation Register Aliasing
- **Issue:** `ldrb`/`strb` instructions were occasionally using 64-bit `x` registers or incorrect aliases, causing assembly errors.
- **Fix:** Introduced `aarch64_get_w_register_name` to ensure byte operations always use 32-bit `w` register names.

### Instruction Ordering and Semantics
- **Issue:** `INS_CMP` was incorrectly translating x86 AT&T syntax (which is `dest - src`) to AArch64, causing reversed conditional jumps.
- **Fix:** Corrected operand order in `archaarch64.c` for `cmp` instructions.

### Numeric Label Standardization
- **Issue:** Inconsistent label generation led to assembly errors or incorrect jumps.
- **Fix:** Standardized on `L<num>` format for all internal labels in the AArch64 backend.

### Secure Label Resolution
- **Issue:** Using `ldr` for label addresses was failing for far labels or incorrect offsets.
- **Fix:** Switched to `adrp` + `add` (via `@PAGE` and `@PAGEOFF`) for robust label address resolution on Darwin.

### Occam Process Return Protocol
- **Issue:** The AArch64 backend used a standard C `ret` instruction for Occam procedure returns, which caused hangs because Occam processes return by jumping to the address stored in `Wptr[Iptr]`.
- **Fix:** Updated `compose_aarch64_return` to emit a jump to `Wptr[Iptr]` (specifically `[x28, #-8]` on 64-bit).

### Large Displacement Support
- **Issue:** AArch64 `ldr`/`str` instructions have a limited immediate offset range (±256 bytes for unscaled). KRoC often generates much larger offsets for workspace variables.
- **Fix:** Implemented `aarch64_emit_mem_op` which automatically detects large displacements and loads them into scratch register `x15` before performing the memory operation.

### Logical Instruction Constants
- **Issue:** `INS_AND`, `INS_OR`, and `INS_XOR` failed when one operand was a constant, as AArch64 logical instructions have strict immediate encoding rules.
- **Fix:** Updated logical instruction handlers to load constants into scratch register `x16` using `aarch64_emit_large_immediate` when they don't fit the immediate encoding.

### Codemap and Dynamic Code Entry Support
- **Issue:** Debugging information and dynamic process loading were broken because `RTL_CODEMAP` and `RTL_DYNCODEENTRY` were not handled in the AArch64 backend.
- **Fix:** Added handlers to `aarch64_code_to_asm_stream` to emit codemap labels, workspace/vectorspace sizes, and type hashes.

### macOS Symbol Mapping Overhaul
- **Issue:** Linker errors occurred because Occam symbols (prefixed with `_O_`, `_C_`, etc.) didn't consistently follow macOS's requirement for a leading underscore, or clashed with procedure names.
- **Fix:** Overhauled `aarch64_convert_symbol_name` to correctly map KRoC internal prefixes (`$`, `^`, `*`, `@`) to macOS-compatible names (`_O_`, `_C_`, `_M_`, `_A_`) while ensuring standard C and kernel symbols bypass these prefixes.

### Annotation Comment Prefixing
- **Issue:** `INS_ANNO` was dumping raw text to the assembly stream, which sometimes contained characters the assembler interpreted as invalid directives.
- **Fix:** Updated the `INS_ANNO` handler to ensure all annotations are prefixed with `//` comment markers.

### Duplicate Symbol Fix for Large Files
- **Issue:** The `defined_labels` array in `archaarch64.c` was too small (1024), causing `tranx86` to lose track of defined labels in large Occam modules and emit duplicate globals.
- **Fix:** Increased `defined_labels` size to 16384.

### Kernel Call Linkage Consistency
- **Issue:** Kernel calls generated by `tranx86` (via `compose_aarch64_kcall`) didn't match the `kernel_` prefix expected by `libccsp`.
- **Fix:** Ensured all kernel entries and jumps are prefixed with `kernel_` and respect the global underscore prefix.

## 2. Runtime (CCSP) Fixes

### Workspace Alignment and Offsets
- **Issue:** The root process was incorrectly calculating its parameter offsets (like `scr`), leading to crashes or "Magic Screen" (shutdown) failures.
- **Fix:** Set the default `ws_adjust` to 0 in `tstate.c` for AArch64.

### 64-bit Word Definition
- **Issue:** `word` was incorrectly defined as a 32-bit integer in some headers.
- **Fix:** Ensured `word` is a true 64-bit integer across `ukcthreads_types.h` and `sched_types.h` for AArch64.

### Stack Alignment and Cachelines
- **Issue:** Misaligned stack pointers or `sched_t` structures caused `SIGBUS` (alignment faults) or `EXC_BAD_ACCESS`.
- **Fix:** 
  - Updated `entry.c` to enforce 64-byte alignment for the kernel stack buffer.
  - Ensured `K_ENTRY` maintains 16-byte `sp` alignment.

### Calling Convention Stability
- **Issue:** Inconsistent parameter passing between the kernel and Occam code.
- **Fix:** Standardized on `x0=param0`, `x1=sched`, `x2=wptr` for kernel entries and ensured `x25` (Sched) and `x28` (Wptr) are correctly restored/initialized.

### Kernel Exit Gracefulness
- **Issue:** The process would occasionally crash with `SIGSEGV` after finishing execution.
- **Fix:** Modified `ccsp_kernel_exit` to call `exit()` directly on AArch64, preventing fall-through in `NO_RETURN` functions.

### Local Forward/Backward Label Collision
- **Issue:** `tranx86` generates local forward and backward jump labels (`ARG_FLABEL`, `ARG_BLABEL`) numbered starting from `0`. In `archaarch64.c`, these were translated strictly as `L<num>`, causing collisions with the global numeric labels generated by `occ21` for procedures (e.g., `L0` for the first procedure). This caused calls to Occam procedures to jump to the wrong locations or corrupt the runtime parameter extraction.
- **Fix:** Implemented `aarch64_get_label_prefix` to correctly differentiate local labels. Global `ARG_LABEL` uses `L<num>`, while local forward jumps use `LF<num>` and backward jumps use `LB<num>`.

### Procedure Return & Magic Screen Desynchronization
- **Issue:** Programs crashed at shutdown (Magic Screen) because the workspace pointer (`x28`) was left unbalanced by `I_RET`. The root `__main` procedure does not allocate an `I_CALL` temporary frame, but its implicit `I_RET` pops 32 bytes anyway, moving the workspace out of bounds and causing garbage parameters (like the `scr` channel) to be read.
- **Fix:** Updated `tstate.c` to set `ws_adjust = 32` for 64-bit platforms (specifically AArch64). This allows the Magic Screen block generated by `etcrtl.c` to subtract the 32 bytes back from `x28` before resolving its parameters, perfectly balancing the `I_RET` stack shift and preventing the `EXC_BAD_ACCESS` / `Bus error`.

### Process Resumption Restoration
- **Issue:** The `save_return` function in `sched.c` was disabled for AArch64 via an `#ifndef`, preventing Occam processes from saving their resumption addresses during context switches.
- **Fix:** Removed the `#ifndef __aarch64__` guard to restore standard process resumption logic.

## 3. Build System and Frontend Fixes

### Hardcoded macOS Version Removal
- **Issue:** Hardcoded `-mmacosx-version-min=11.0` in `occbuild` and several module files caused the linker to ignore object files built for the actual host version (e.g., macOS 15+), leading to "ignored file" warnings and linkage failures.
- **Fix:** Removed the hardcoded version strings from `tools/kroc/occbuild.in` and related `.module` files to allow the system default (or user-specified version) to take precedence.

### External STUB Symbol Conversion
- **Issue:** Occam modules calling external C functions (stubs) were failing to link because the stub names (e.g., `socket_accept`) were not being passed through the architecture's symbol converter, missing the required leading underscore on macOS.
- **Fix:** Updated `generate_call` in `tools/tranx86/etcrtl.c` to use `make_c_name` for all stub symbols.
